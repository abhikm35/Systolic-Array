# End-to-End Run Walkthrough: Matrix Multiplication Example

This document traces one full run using **concrete matrices** and points to the **exact files and lines** where each step happens.

---

## Example: 4×4 matrix multiply

We use the data generated by `scripts/gen_test_data.py` (seed 42):

**A (inputs/activations), row-major:**
```
[ 3  0  1  3]
[-1  1  1  3]
[-2 -1  3 -1]
[-1  1  0 -1]
```

**B (weights):**
```
[ 2  1 -2  0]
[ 2  2 -2  0]
[ 1 -3  0 -2]
[ 2  1  0 -3]
```

**Golden C = A × B (row-major):**
```
[13   3  -6 -11]
[ 7   1   0 -11]
[-5 -14   6  -3]
[-2   0   0   3]
```

**Instruction stream:** `[4, 4, 4, 0]` → one MMM: A is 4×4, B is 4×4, C is 4×4; then 0 = end.

---

# Phase 1: Generate test data (Python)

**Where:** `scripts/gen_test_data.py`

1. **Create matrices A and B**  
   - **Lines 54–55:** `A = np.random.randint(-3, 4, (N, N))` and same for `B`.  
   - With seed 42 this produces the A and B above.

2. **Compute golden output C**  
   - **Line 60:** `C = A @ B` (NumPy matrix multiply).  
   - **Lines 61–62:** Flatten C row-major and clamp to 16-bit: `golden_O = [to_s16(x) for x in C_flat]`.  
   - This is the reference the testbench will compare against.

3. **Build instruction stream**  
   - **Lines 67–68:** `instr = [N, N, N, 0]` → `[4, 4, 4, 0]`, then `instr_hex` for file output.

4. **Layout for RTL**  
   - **Lines 76–81:** A flattened row-major (`order="C"`), B flattened column-major (`order="F"`) so the file order matches how the RTL reads B (column-by-column).

5. **Write hex files**  
   - **Lines 87–90:** `write_hex_lines(out_dir / "memA.hex", A_hex)` and similarly for `memB.hex`, `memI.hex`, `golden_O.hex`.  
   - **Lines 36–41:** Each value is one line, 4 hex digits (e.g. `0003`, `FFFE` for -2).

6. **Summary file**  
   - **Lines 94–105:** Write `tb/data/summary.txt` with A, B, C, and instructions.

**Result:** `tb/data/memA.hex`, `memB.hex`, `memI.hex`, `golden_O.hex`, `summary.txt` exist and are used by the testbench.

---

# Phase 2: Simulation starts; testbench loads files

**Where:** `tb/tb_top.sv`

1. **Clock and reset**  
   - **Lines 44–45:** `initial clk = 0`; `always #5 clk = ~clk` (10 time units period).  
   - **Lines 47–51:** `rst_n = 0` for 3 cycles, then `rst_n = 1`.

2. **Load Python-generated files into TB arrays**  
   - **Lines 61–66:** In an `initial` block, `$readmemh("tb/data/memA.hex", memA_file)` and same for `memB_file`, `memI_file`, `golden_O`.  
   - So `memA_file[0]=3`, `memA_file[1]=0`, … `memA_file[15]=-1` (row-major A).  
   - `golden_O[0]=13`, `golden_O[1]=3`, … `golden_O[15]=3` (row-major C).

3. **DUT connection**  
   - **Lines 71–96:** Instance `top` as `u_dut` with `clk`, `rst_n`, `ap_start`, `ap_done`, and all memory ports (`addrA`/`dataA`/`enA`, etc.) wired to TB signals.

---

# Phase 3: Load A, B, and I into the RTL

**Where:** testbench drives ports; RTL writes internal memories.

1. **Wait for reset release**  
   - **tb/tb_top.sv lines 111–112:** `@(posedge rst_n); repeat (2) @(posedge clk);`

2. **Load matrix A into DUT**  
   - **tb/tb_top.sv lines 115–120:** For `load_addr = 0` to `N_SQ-1` (15), each cycle: `addrA = load_addr`, `dataA = memA_file[load_addr]`, `enA = 1`.  
   - So cycle-by-cycle: addr 0 → 3, addr 1 → 0, addr 2 → 1, … addr 15 → -1.

   **RTL response (top):**  
   - **rtl/top.sv lines 76–82:** `always_ff @(posedge clk)`: when `enA`, `memA[addrA] <= dataA`.  
   - So after the loop, `memA[0..15]` holds A in row-major order.

3. **Load matrix B into DUT**  
   - **tb/tb_top.sv lines 123–129:** Same pattern: `addrB`/`dataB`/`enB` from `memB_file[0..15]` (column-major in file).  
   - **rtl/top.sv lines 84–89:** When `enB`, `memB[addrB] <= dataB`.  
   - So `memB[0..15]` holds B in the column-major order the controller will use.

4. **Load instruction memory**  
   - **tb/tb_top.sv lines 132–139:** For `load_addr = 0..` until `memI_file[load_addr] == 0`: drive `addrI`, `dataI`, `enI`.  
   - So addresses 0,1,2,3 get 4, 4, 4, 0.  
   - **rtl/top.sv lines 93–98:** When `enI`, `memI[addrI] <= dataI`.  
   - So `memI[0]=4`, `memI[1]=4`, `memI[2]=4`, `memI[3]=0`.

---

# Phase 4: Start accelerator and controller FSM

**Where:** `tb/tb_top.sv` pulses `ap_start`; `rtl/controller.sv` runs the FSM.

1. **Pulse ap_start**  
   - **tb/tb_top.sv lines 144–147:** One cycle `ap_start = 1`, then `ap_start = 0`.

2. **Controller leaves IDLE**  
   - **rtl/controller.sv lines 160–164:** In `S_IDLE`, when `ap_start`, `next_state = S_FETCH_A`.  
   - **Lines 81–82:** `state <= next_state` on the next clock, so state becomes `S_FETCH_A`.

3. **Fetch (a, b, c) from instruction memory**  
   - **rtl/top.sv line 124:** `instr_addr` is driven by the controller; **line 122:** `instr_dout = memI[instr_addr]` (combo read).  
   - **rtl/controller.sv line 116:** `instr_addr = instr_ptr`; initially `instr_ptr = 0` (lines 77, 90).  
   - **S_FETCH_A (lines 166–173):** `instr_dout = memI[0] = 4` (non-zero), so `next_state = S_FETCH_B`.  
   - **Lines 96–98:** On transition to S_FETCH_B, `a_reg <= instr_dout` (4), `instr_ptr <= 1`.  
   - **S_FETCH_B (lines 176–178):** `next_state = S_FETCH_C`. **Lines 100–102:** `b_reg <= memI[1]` (4), `instr_ptr <= 2`.  
   - **S_FETCH_C (lines 181–183):** `next_state = S_LOAD`. **Lines 104–106:** `c_reg <= memI[2]` (4), `instr_ptr <= 3`.

4. **S_LOAD: clear array and start run**  
   - **rtl/controller.sv lines 186–192:** `sa_clear = 1'b1`, `sa_start = 1'b1`, `addrA_r = 0`, `addrB_r = 0`, `next_state = S_RUN`.  
   - **Lines 110–113:** On the same transition, `run_cnt <= 0`, `out_cnt <= 0`.  
   - **rtl/top.sv:** `sa_clear` and `sa_start` go to `u_systolic_array` (lines 191–192).  
   - **rtl/systolic_array.sv lines 54–57:** When `clear`, `cycle_cnt`/`out_idx`/`running` reset; **lines 58–60:** when `start && !running`, `running <= 1`, `cycle_cnt <= 0`.

---

# Phase 5: Feed phase – addresses and data into the array

**Where:** Controller is in `S_RUN`; it drives read addresses and feeds the systolic array.

**rtl/controller.sv lines 194–216:** In `S_RUN`:

- **Line 116:** `instr_addr = instr_ptr` (still 3; not used for data).  
- **Lines 197–198:**  
  - `addrA_r = run_cnt` (0, 1, 2, …, 15 then keeps increasing).  
  - `addrB_r = (run_cnt % N)*N + (run_cnt / N)` → column-major: 0,4,8,12, 1,5,9,13, …  
- **rtl/top.sv lines 121–122:** `dataA_r = memA[addrA_r]`, `dataB_r = memB[addrB_r]` (combo).  
- **Lines 201–206:** While `run_cnt < N_SQ` (16), `sa_in_a = dataA_r`, `sa_in_b = dataB_r`; after that, both forced to 0 so the drain phase doesn’t add garbage.

So for the first 16 cycles of S_RUN:

| run_cnt | addrA_r | addrB_r | sa_in_a (memA) | sa_in_b (memB) |
|---------|---------|---------|----------------|----------------|
| 0       | 0       | 0       | A[0,0]=3       | B[0,0]=2       |
| 1       | 1       | 4       | A[0,1]=0       | B[1,0]=2       |
| 2       | 2       | 8       | A[0,2]=1       | B[2,0]=1       |
| 3       | 3       | 12      | A[0,3]=3      | B[3,0]=2       |
| 4       | 4       | 1       | A[1,0]=-1      | B[0,1]=1       |
| …       | …       | …       | …              | …              |
| 15      | 15      | 11      | A[3,3]=-1      | B[3,2]=0       |

**rtl/systolic_array.sv lines 95–97:** Only row 0 and column 0 get data: `a_bus[0][0] = sa_in_a`, `b_bus[0][0] = sa_in_b`. So in this design only PE(0,0) sees the full streams; other PEs get zeros on their A/B inputs.

---

# Phase 6: Computation in the PEs

**Where:** `rtl/pe.sv`

**Lines 23–38:** Every cycle (when not in reset):

- **Lines 30–31:** `out_a <= in_a`, `out_b <= in_b` (pass-through to the next PE).  
- **Lines 33–36:** If `clear`, `acc_out <= 0`; else `acc_out <= acc_out + (in_a * in_b)`.

So each PE does a running sum of products. PE(0,0) receives the full A row 0 and B column 0 streams and computes C[0,0] = 3×2 + 0×2 + 1×1 + 3×2 = 6+0+1+6 = 13. Other PEs in this design only get one non-zero stream, so their accumulators don’t match the full dot product (architectural limitation of the current single-feed design).

**rtl/systolic_array.sv lines 104–122:** The 4×4 PE grid is wired so `a_bus` flows left-to-right and `b_bus` top-to-bottom; each PE’s `acc_out` drives `acc_grid[i][j]`.

---

# Phase 7: Drain phase and write-back to memO

**Where:** `rtl/systolic_array.sv` (cycle counter, drain); `rtl/controller.sv` (writes to O).

1. **Array cycle counter**  
   - **rtl/systolic_array.sv lines 60–65:** While `running`, `cycle_cnt` increments each cycle.  
   - **Lines 38–41:** `FEED_CYCLES = 7`, `DRAIN_CYCLES = 16`, `TOTAL_CYCLES = 23`.  
   - **Line 68:** `drain_phase = (cycle_cnt >= FEED_CYCLES)`. So after cycle 7 we enter drain.

2. **Output index and data**  
   - **Lines 62–64:** In drain, `out_idx` increments each cycle (0..15).  
   - **Lines 69–70:** `sa_out_valid = drain_phase && (out_idx < DRAIN_CYCLES)`. So 16 cycles of valid output.  
   - **Lines 72–75:** `out_row = out_idx / N`, `out_col = out_idx % N`; **line 75:** `sa_out_data = acc_grid[out_row][out_col]` (row-major order).

3. **sa_done**  
   - **Line 70:** `sa_done = (cycle_cnt >= TOTAL_CYCLES - 1)` → true when `cycle_cnt == 22`.

4. **Controller writes to memO**  
   - **rtl/controller.sv lines 208–212:** When `sa_out_valid`, `weO = 1`, `addrO_w = baseO + out_cnt`, `dataO_w = sa_out_data[15:0]`.  
   - **Lines 117–120:** Each time `sa_out_valid`, `out_cnt` increments on the next clock.  
   - **rtl/top.sv lines 132–136:** When `weO`, `memO[addrO_w] <= dataO_w`.  
   - `baseO` starts at 0 (controller line 90), so results go to `memO[0..15]` in row-major order.

5. **Transition to S_WRITE_BACK and then S_FETCH_A**  
   - **rtl/controller.sv line 214:** When `sa_done`, `next_state = S_WRITE_BACK`.  
   - **Lines 219–222:** In `S_WRITE_BACK`, `next_state = S_FETCH_A`. So we go back to fetch the next instruction word.

6. **Next instruction is 0 → S_DONE**  
   - **rtl/controller.sv line 116:** `instr_addr = instr_ptr` (3).  
   - **rtl/top.sv line 122:** `instr_dout = memI[3] = 0`.  
   - **Lines 166–170:** In `S_FETCH_A`, when `instr_dout == 0`, `next_state = S_DONE`.  
   - **Lines 224–228:** In `S_DONE`, `ap_done = 1'b1`; when `!ap_start`, `next_state = S_IDLE`.  
   - **rtl/top.sv line 33:** `ap_done` is an output of the controller, so the top-level `ap_done` goes high.

---

# Phase 8: Testbench reads memO and compares to golden

**Where:** `tb/tb_top.sv`

1. **Wait for ap_done**  
   - **Lines 150–151:** `wait (ap_done);` then a couple of cycles.

2. **Read O and compare**  
   - **Lines 155–163:** For `i = 0` to `N_SQ-1`: set `addrO = i`, wait one cycle (sync read), then compare `dataO` to `golden_O[i]`.  
   - **rtl/top.sv lines 104–110:** `dataO <= memO[addrO]` on the clock edge, so the value appears one cycle after `addrO` is set.  
   - So for each index we get the value that was written during the drain phase (or 0 if not written). Any mismatch is printed and `err_count` is incremented.

3. **Pass/fail**  
   - **tb/tb_top.sv lines 167–170:** If `err_count == 0`, print PASS; else print FAIL and the number of mismatches. **Line 171:** `$finish`.
